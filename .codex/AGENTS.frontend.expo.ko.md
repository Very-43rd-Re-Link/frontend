# AGENTS.md — React Native + Expo 프론트엔드용

> 목적: Codex가 React Native + Expo 프론트엔드 저장소에서 일관되게 작업하도록 하는 지침입니다.  
> 범위: 이 파일이 있는 디렉터리와 하위 디렉터리에 적용합니다. 더 가까운 하위 디렉터리에 별도 `AGENTS.md`가 있으면 그 지침을 우선합니다.

---

## 0. 기본 운영 원칙

너는 이 저장소 안에서 작업하는 자율 코딩 에이전트다.

우선순위는 다음과 같다.

1. 사용자가 명시적으로 요구하지 않은 기존 동작은 보존한다.
2. 문제를 해결하는 가장 작고 안전한 변경을 선호한다.
3. 영리하지만 복잡한 코드보다 읽기 쉽고 유지보수하기 쉬운 코드를 우선한다.
4. 변경 후 가능한 범위에서 타입 체크, 린트, 테스트, 빌드로 검증한다.
5. 여러 방법이 가능하면 장단점을 짧게 설명하고, 가장 보수적인 선택을 기본으로 한다.
6. 불확실한 부분은 숨기지 말고, 어떤 가정을 했는지 명확히 말한다.
7. Expo/React Native에서 자동 생성된 네이티브 파일은 기본적으로 수정하지 않는다.

사용자가 “구현”을 요청하면 구현한다.  
사용자가 “조사”를 요청하면 먼저 조사하고, 불필요한 수정을 하지 않는다.  
사용자가 “리팩토링”을 요청하면 공개 동작과 API 호환성을 보존한다.

## 1. 응답 스타일

기본 응답은 간결하고 실용적인 한국어로 작성한다.

사용자에게 보이는 모든 메시지, 커밋 메시지, PR 설명, 이슈 본문, 작업 로그는 한국어로 작성한다.

작업 결과 보고는 가능하면 아래 형식을 따른다.

1. 변경한 내용
2. 변경 이유
3. 검증 방법 및 결과
4. 남은 위험 또는 후속 작업

버그를 보고할 때는 다음을 포함한다.

- 원인
- 정확한 위치
- 최소 수정 방법
- 검증 방법

장황한 이론 설명은 사용자가 요청하지 않는 한 피한다.

---

## 2. 작업 전 확인

파일을 수정하기 전에 다음을 확인한다.

1. 저장소 구조를 먼저 훑는다.
2. `package.json`, `app.json` 또는 `app.config.*`, `babel.config.*`, `tsconfig.json`, `eas.json`을 확인한다.
3. Expo SDK 버전, React Native 버전, 패키지 매니저를 확인한다.
4. 기존 폴더 구조, 네이밍, 컴포넌트 패턴, 스타일링 방식을 확인한다.
5. 비슷한 구현이 있는지 검색하고, 기존 스타일을 우선 재사용한다.
6. 대상 파일이 수동 관리 파일인지, 자동 생성 파일인지 확인한다.
7. 가능하면 `git status`로 사용자의 미커밋 변경사항을 확인한다.

기존 패턴으로 해결 가능한 문제에 새로운 아키텍처, 상태 관리 라이브러리, UI 라이브러리, 빌드 도구를 도입하지 않는다.

---

## 3. Expo / React Native 핵심 규칙

Expo 프로젝트에서는 다음을 우선한다.

- `app.json`, `app.config.js`, `app.config.ts`, Expo config plugin, dependency 설정으로 해결 가능한 문제는 이 방식으로 해결한다.
- `android/`, `ios/` 아래 생성 파일은 기본적으로 직접 수정하지 않는다.
- 네이티브 파일 수정은 사용자가 명시적으로 요청했거나, 프로젝트가 prebuild 결과물을 의도적으로 관리하는 경우에만 고려한다.
- Android와 iOS 동작 차이를 항상 고려한다.
- Expo Go, development build, EAS Build의 차이를 구분해서 설명한다.
- 네이티브 모듈이 필요한 라이브러리는 Expo Go에서 동작하지 않을 수 있음을 확인한다.
- 권한 설정은 플랫폼별 설정을 확인한다.
  - iOS: `Info.plist`에 반영되는 Expo config
  - Android: `AndroidManifest.xml`에 반영되는 Expo config
- 앱 아이콘, 스플래시, 딥링크, 스킴, 번들 ID, 패키지명 변경은 빌드 영향이 있으므로 반드시 영향 범위를 설명한다.

---

## 4. 생성 파일과 빌드 산출물

다음 파일과 디렉터리는 기본적으로 수정하지 않는다.

- `node_modules/`
- `.expo/`
- `dist/`
- `build/`
- `coverage/`
- `android/`의 자동 생성 또는 prebuild 산출물
- `ios/`의 자동 생성 또는 prebuild 산출물
- generated API client
- compiled assets
- lockfile은 의존성 변경이 필요한 경우에만 수정한다.

Expo/React Native 이슈를 해결할 때는 다음 순서를 선호한다.

1. Expo config 수정
2. config plugin 사용 또는 수정
3. 의존성 버전 정합성 수정
4. 빌드 명령 또는 개발 환경 설정 수정
5. 필요한 경우에만 네이티브 파일 수정

## 5. 코드 변경 규칙

모든 코드 변경은 다음을 따른다.

- diff를 작고 집중적으로 유지한다.
- 관련 없는 리포맷팅을 하지 않는다.
- 기능 작업과 대규모 리팩토링을 섞지 않는다.
- 파일, 컴포넌트, 함수, API 이름 변경은 필요한 경우에만 한다.
- 단순히 코드를 다시 설명하는 주석은 추가하지 않는다.
- 주석은 비즈니스 규칙, 플랫폼 차이, 우회 이유, 비명시적 제약을 설명할 때만 추가한다.
- 축약어보다 명확한 이름을 선호한다.
- 깊은 중첩보다 early return을 선호한다.
- 상태는 가능한 한 가까운 곳에 둔다.
- 전역 상태는 실제 공유 필요가 있을 때만 사용한다.

공개 API, 인증, 결제, 개인정보, 외부 연동, 라우팅 구조, 앱 빌드 설정을 건드릴 때는 더 보수적으로 작업하고 영향 범위를 설명한다.

---

## 6. TypeScript 규칙

TypeScript를 사용하는 프로젝트에서는 다음을 따른다.

- `any`는 피하고, 불가피하면 이유가 명확해야 한다.
- API 응답, 요청 DTO, 화면 파라미터, navigation params는 명시적으로 타입을 둔다.
- null/undefined 가능성을 무시하지 않는다.
- 타입 에러를 `as unknown as ...`로 덮지 않는다.
- 공통 타입은 기존 위치를 우선 따르고, 새 위치를 만들 때는 사용 범위가 명확해야 한다.
- enum보다 union literal type이 기존 코드 스타일에 맞으면 이를 선호한다.
- 컴포넌트 props 타입은 재사용성과 가독성을 기준으로 분리한다.

---

## 7. 컴포넌트 규칙

React / React Native 컴포넌트는 다음을 따른다.

- 함수형 컴포넌트와 hooks를 사용한다.
- 컴포넌트는 작고 명확한 책임을 갖게 한다.
- 재사용이 실제로 발생할 때만 hook으로 추출한다.
- 불필요한 `useMemo`, `useCallback`, `memo`를 남발하지 않는다.
- 리스트에는 안정적인 key를 사용한다.
- loading, empty, error 상태를 함께 고려한다.
- 접근성을 고려한다.
  - 누를 수 있는 요소는 의미 있는 label을 갖게 한다.
  - 텍스트 크기, 터치 영역, 대비를 고려한다.
- 플랫폼별 분기가 필요하면 `Platform.select` 또는 파일 분리(`*.ios.tsx`, `*.android.tsx`)를 검토한다.
- 스타일 시스템이 있다면 이를 우선 사용하고, 임의 inline style 남발을 피한다.

---

## 8. 화면 / 라우팅 규칙

라우팅은 기존 라이브러리와 구조를 따른다.

- Expo Router를 쓰는 프로젝트는 `app/` 디렉터리 규칙을 따른다.
- React Navigation을 쓰는 프로젝트는 기존 navigator 구조를 따른다.
- 화면 파일은 기존 네이밍과 위치를 따른다.
- navigation param은 타입을 명확히 한다.
- 인증이 필요한 화면과 공개 화면의 경계를 임의로 바꾸지 않는다.
- 딥링크, 탭, 스택, 모달 구조 변경은 사용자 흐름에 영향이 있으므로 설명한다.
- 뒤로가기, safe area, keyboard avoiding, header 처리까지 확인한다.

---

## 9. 상태 관리 규칙

상태 관리는 가장 단순한 범위부터 선택한다.

선호 순서:

1. 컴포넌트 local state
2. custom hook
3. Context
4. 기존 전역 상태 라이브러리
5. 새 전역 상태 라이브러리 도입

다음은 피한다.

- 화면 하나에서만 쓰는 값을 전역 상태로 올리기
- 서버 상태와 클라이언트 상태를 혼동하기
- API 캐시를 수동 전역 상태로 중복 관리하기
- 상태 변경 흐름을 여러 파일에 불필요하게 분산하기

React Query, TanStack Query, SWR 등이 이미 있으면 서버 상태는 해당 패턴을 따른다.

---

## 10. API 통신 규칙

API 연동 시 다음을 따른다.

- 기존 API client, fetcher, axios instance를 먼저 확인한다.
- base URL, timeout, headers, auth token 처리는 기존 설정을 따른다.
- 요청/응답 타입을 명시한다.
- 에러 응답 형태를 확인하고 사용자에게 안전한 메시지로 변환한다.
- 토큰, Authorization header, 개인정보를 로그로 남기지 않는다.
- 네트워크 실패, 타임아웃, 서버 에러를 구분할 수 있으면 구분한다.
- 같은 API 호출 로직을 화면마다 중복 작성하지 않는다.
- 백엔드 API 계약을 임의로 바꾸지 않는다.

---

## 11. 인증 / 소셜 로그인 규칙

인증 관련 변경은 특히 보수적으로 한다.

- access token, refresh token 저장 위치는 기존 정책을 따른다.
- 민감 토큰은 `expo-secure-store` 또는 프로젝트가 선택한 보안 저장소를 사용한다.
- AsyncStorage에 민감 토큰을 저장하지 않는다. 기존 프로젝트가 그렇게 하고 있다면 위험을 설명한다.
- Google, Kakao, Apple 로그인은 “클라이언트 토큰 획득”과 “백엔드 검증/로그인”을 분리해서 생각한다.
- Apple 로그인은 iOS, 실제 기기, bundle identifier, redirect 설정 제약을 고려한다.
- 로컬 테스트가 어려운 소셜 로그인은 mock, dev endpoint, 테스트 계정 전략을 제안한다.
- 로그아웃은 로컬 토큰 삭제, 서버 세션/refresh token 무효화 여부를 구분한다.

---

## 12. 폼 / 입력 / 검증 규칙

사용자 입력은 경계에서 검증한다.

- 기존 폼 라이브러리와 validation 패턴을 따른다.
- 빈 값, 길이 제한, 형식 오류, 서버 검증 실패를 처리한다.
- 키보드에 가려지는 입력 영역을 고려한다.
- 모바일 환경에서 입력 타입, 자동완성, 보안 입력 여부를 확인한다.
- 검증 메시지는 사용자가 이해할 수 있게 작성한다.

---

## 13. UI / 스타일 규칙

UI 변경은 기존 디자인 시스템을 우선한다.

- 기존 컴포넌트, 색상 토큰, spacing, typography를 재사용한다.
- 새 UI 라이브러리는 승인 없이 추가하지 않는다.
- 다크모드 지원 여부를 확인한다.
- SafeArea를 고려한다.
- 터치 영역은 모바일 사용성을 고려한다.
- 로딩, 빈 상태, 오류 상태, 권한 거부 상태를 함께 설계한다.
- 이미지와 아이콘은 기존 asset 관리 방식을 따른다.
- 화면 깨짐을 막기 위해 작은 화면, 긴 텍스트, 다른 폰트 크기를 고려한다.

---

## 14. 권한 / 네이티브 기능 규칙

카메라, 위치, 알림, 연락처, 파일, 사진, 마이크 등 권한이 필요한 기능은 다음을 확인한다.

- Expo SDK에서 제공하는 API인지 확인한다.
- iOS/Android 권한 문구와 설정이 필요한지 확인한다.
- 권한 거부, 다시 묻기 불가, 설정 이동 시나리오를 처리한다.
- 실제 기기에서만 가능한 기능인지 확인한다.
- 백그라운드 동작은 플랫폼 제약과 스토어 심사 리스크를 고려한다.
- 푸시 알림은 Expo Push Token, FCM/APNs, 백엔드 저장 흐름을 구분한다.

## 15. 성능 규칙

성능 최적화 전에는 병목을 먼저 확인한다.

자주 확인할 부분:

- FlatList/SectionList key와 renderItem 안정성
- 큰 이미지 최적화
- 불필요한 재렌더링
- 과도한 전역 상태 구독
- 반복 네트워크 요청
- 화면 진입 시 무거운 동기 작업
- 너무 큰 번들
- 개발 빌드와 프로덕션 빌드 차이

최적화는 단순한 것부터 적용하고, 성능 개선을 위해 코드 구조를 과도하게 복잡하게 만들지 않는다.

## 16. 보안 / 개인정보 규칙

다음은 절대 로그로 남기거나 응답에 노출하지 않는다.

- access token
- refresh token
- authorization header
- 비밀번호
- 소셜 로그인 provider token
- 개인 식별 정보 전체
- 위치 정보 원문
- 결제 정보
- `.env` 값
- 운영 서버 URL 중 민감한 내부 주소

보안 관련 변경 시 확인할 것:

- 인증 여부
- 권한 분기
- 토큰 저장 방식
- 민감정보 마스킹
- API 에러 메시지 노출
- 딥링크 open redirect 가능성
- WebView 사용 시 origin, injected script, navigation 제한

## 17. 의존성 정책

새 의존성을 추가하기 전에는 다음을 확인한다.

1. 이미 같은 역할의 라이브러리가 있는가?
2. React Native/Expo SDK와 호환되는가?
3. Expo Go에서 동작하는가, development build가 필요한가?
4. iOS/Android 모두 지원하는가?
5. 유지보수가 활발한가?
6. 번들 크기와 네이티브 설정 부담이 적절한가?

사소한 유틸 함수 때문에 새 라이브러리를 추가하지 않는다.  
관련 없는 의존성 업그레이드는 하지 않는다.

## 18. 테스트 / 검증

변경 후 가능한 가장 관련성 높은 검증을 수행한다.

선호 순서:

1. 변경한 로직의 단위 테스트
2. 관련 컴포넌트 테스트
3. TypeScript 체크
4. Lint
5. Expo start 또는 빌드 확인
6. Android/iOS 실행 확인
7. 수동 추론

대표 명령은 프로젝트에 존재하는 스크립트를 기준으로 사용한다.

예시:

```bash
npm run typecheck
npm run lint
npm test
npx expo start
npx expo run:android
npx expo run:ios
eas build --platform android
eas build --platform ios
```

실제 명령은 `package.json`과 프로젝트 설정을 확인한 뒤 선택한다.

검증을 실행할 수 없으면 이유와 사용자가 실행할 정확한 명령을 남긴다.

---

## 19. 디버깅 규칙

디버깅할 때는 다음 순서를 따른다.

1. 정확한 에러 메시지를 읽는다.
2. 실패하는 최소 경계를 찾는다.
3. 증상에서 원인 방향으로 추적한다.
4. 임시 우회보다 근본 원인을 우선 수정한다.
5. Expo, Metro, Gradle, CocoaPods, Xcode, Android SDK 중 어느 계층 문제인지 구분한다.
6. 캐시 삭제는 마지막 수단으로 사용하고, 왜 필요한지 설명한다.
7. 수정 후 가장 좁은 검증 명령을 실행한다.

자주 구분할 문제:

- Java/JDK 버전 문제
- Gradle/Android Gradle Plugin 버전 문제
- Expo SDK와 라이브러리 버전 불일치
- Metro 캐시 문제
- iOS CocoaPods 문제
- native module이 Expo Go에서 동작하지 않는 문제
- emulator/simulator 환경 문제
- 환경변수 또는 `.env` 로딩 문제

## 20. Git 규칙

작업 전 가능하면 `git status`를 확인한다.

- 사용자의 미커밋 변경사항을 덮어쓰지 않는다.
- 사용자가 명시하지 않으면 commit, push, merge, rebase를 하지 않는다.
- 충돌이나 예상치 못한 변경이 있으면 먼저 설명한다.
- 변경 범위는 요청된 작업에 한정한다.
- 브랜치를 만들 때는 `{git flow}/#{issue-number}-{작업 내용 영어로}` 형식을 따른다.
  - 예: `feature/#123-login-screen`
  - 작업 내용은 짧은 영어 소문자와 하이픈으로 작성한다.

커밋 메시지를 작성해야 할 때는 다음 형식을 선호한다.

```text
type: 간결한 변경 요약
```

대표 type:

- `feat`
- `fix`
- `refactor`
- `test`
- `docs`
- `chore`
- `build`
- `ci`

PR을 작성하거나 요약할 때는 다음을 따른다.

- `.github/pull_request_template.md`를 사용한다.
- 적용 가능한 모든 섹션을 한국어로 채운다.
- 관련 이슈 번호를 알고 있으면 포함한다.

이슈를 작성할 때는 다음을 따른다.

- `.github/ISSUE_TEMPLATE/` 아래의 적절한 템플릿을 사용한다.
- 이슈 제목과 본문은 한국어로 작성한다.
- 선택한 템플릿 구조는 유지한다.

## 21. 문서화 규칙

동작, 설정, 명령어, API 사용 방식이 바뀌면 문서를 갱신한다.

좋은 문서는 다음을 만족한다.

- 짧다.
- 정확하다.
- 코드 가까이에 있다.
- 당연한 구현 설명보다 사용법과 결정 이유를 설명한다.

작은 내부 변경 때문에 광범위한 문서를 새로 만들지 않는다.

---

## 22. 리뷰 모드

코드 리뷰를 요청받으면 다음 순서로 본다.

1. 정확성
2. 보안
3. 개인정보/토큰 노출
4. 데이터 손실 가능성
5. 깨지는 변경
6. 플랫폼별 문제
7. 성능 저하
8. 유지보수성
9. 스타일 일관성

각 이슈는 다음 형식으로 작성한다.

- Severity: `critical`, `major`, `minor`, `nit`
- Location
- Problem
- Suggested fix

사소한 취향 차이는 코드가 충분히 깨끗할 때만 언급한다.

---

## 23. 리팩토링 모드

리팩토링 시 다음을 따른다.

- 동작을 보존한다.
- diff를 리뷰 가능하게 유지한다.
- 이름, 경계, 중복, 테스트 가능성을 개선한다.
- 외부 API, 라우팅, 저장소 구조, 빌드 설정은 요청 없이는 바꾸지 않는다.
- 가능하면 테스트를 추가하거나 기존 테스트를 유지한다.

대규모 재작성보다 점진적 리팩토링을 선호한다.

---

## 24. PR 요약 템플릿

GitHub PR 본문이 아닌 일반 변경 요약을 요청받으면 다음 형식을 사용한다.

```markdown
## 요약

- 
- 

## 검증

- [ ] 명령 또는 확인 사항

## 참고

- 
```

검증을 실행하지 못했다면 다음처럼 쓴다.

```markdown
## 검증

- 실행하지 않음: <사유>
```

실제 PR 본문을 작성할 때는 이 일반 요약 형식이 아니라 `.github/pull_request_template.md`를 사용한다.

## 25. 완료 전 체크리스트

응답을 마치기 전에 확인한다.

- 무엇을 변경했는가?
- 어떤 파일을 변경했는가?
- 어떤 검증을 했는가?
- 검증하지 못했다면 이유는 무엇인가?
- 남은 위험이나 가정은 무엇인가?
- Expo/React Native 생성 파일을 수정하지 않았는가?
- 민감정보를 출력하지 않았는가?

파일을 변경하지 않았다면 “파일 변경 없음”이라고 말한다.  
작업이 불완전하면 무엇이 빠졌고 왜 빠졌는지 명확히 설명한다.
